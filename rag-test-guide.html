Deep Dive — Each Test Type
Failure mode, technique, step-by-step flow, concrete example, and the actual assertion
🧪
01

Faithfulness

LLM-as-Judge + Ragas
Failure Mode
The LLM answer contains facts that came from its training data, not from the retrieved chunks. Even if the answer is factually correct, it's untestable — you can't tell where the fact came from.
How It Works
1
Call /query → get answer + retrieved chunks
2
Ask LLM judge: "List every claim in the answer. For each claim, find supporting evidence in these chunks."
3
Score = supported_claims / total_claims
4
Also send to Ragas /evaluate for standardised NLI-based score (comparable across teams)
Concrete Example
Query "What is the minimum password length at Solaris Tech?"
Chunk "Passwords must be a minimum of 16 characters…"
Answer ✓ "16 characters" → supported by chunk → score: 1.0
Answer ✗ "12 characters (industry standard)" → not in chunk → hallucination
Key Assertion
expect(score).toBeGreaterThanOrEqual(0.85); // Ragas threshold: at most 1 in 7 claims unsupported
✓ Pass
All claims trace back to a chunk sentence
✗ Fail
Answer adds facts not present in any chunk
🎯
02

Context Precision

LLM-as-Judge (per chunk)
Failure Mode
The top-K retrieved chunks are irrelevant to the query. Even a perfect LLM can't produce a good answer from the wrong context — garbage in, garbage out.
How It Works
1
Call /query with topK=3 → get 3 chunks
2
For each chunk: ask LLM judge "Score 0–1 how relevant this chunk is to the query. Give a reason."
3
Count chunks with score ≥ 0.7
4
Assert: at least N of K chunks are relevant
Concrete Example
Query "What are the password requirements?"
Chunk 1 ✓ security policy doc → relevance: 0.98 → relevant
Chunk 2 ✓ password change cadence → relevance: 0.85 → relevant
Chunk 3 ✗ vacation policy doc → relevance: 0.12 → irrelevant
Precision 2/3 = 0.67 → PASS (≥ 2 required)
Key Assertion
const relevantCount = scored.filter(s => s >= 0.7).length; expect(relevantCount).toBeGreaterThanOrEqual(MIN_RELEVANT_CHUNKS);
✓ Pass
Most retrieved chunks are topically relevant to the query
✗ Fail
Retriever pulled in off-topic chunks — embedding alignment issue
🚫
03

Negative (Hallucination Detection)

Phrase Matching
Failure Mode
A confident answer to a question that isn't in the KB at all. The LLM uses its training data to fill the gap. In compliance / legal / medical contexts this is dangerous — the user trusts the answer.
How It Works
1
Load out-of-KB questions from test-cases.json (topics not in any document)
2
Call /query for each question
3
Check if the answer contains any configured refusal phrase (e.g. "I don't have information", "not covered in", "I cannot")
4
No refusal phrase = hallucination detected = FAIL
Concrete Example
Query "How many vacation days do Solaris Tech employees get?" (NOT in KB)
Answer ✓ "I don't have information about vacation days in the knowledge base."
Answer ✗ "Employees typically receive 15 days per year." (from training data — hallucination)
Key Assertion
function containsRefusal(text) { return REFUSAL_PHRASES.some(p => text.includes(p)); } expect(containsRefusal(data.answer)).toBe(true);
✓ Pass
System politely declines — "I don't have that information"
✗ Fail
System invents an answer from training data — live hallucination caught
🔍
04

Retrieval Unit Test

Deterministic Source ID Match
Failure Mode
After swapping embedding models, changing chunk size, or re-indexing documents, a known query no longer retrieves the correct document. The pipeline is silently degraded.
How It Works
1
Define ground-truth pairs in test-cases.json: query → expected doc ID
2
Call /query with topK=3
3
Extract the source field from each retrieved chunk
4
Assert: expected doc ID appears anywhere in the top-K list
Concrete Example
Query "What is the minimum password length?"
Expected doc-003-security must appear in top-3
Got ✓ [doc-003-security(0.94), doc-001-vacation(0.41), doc-002(0.38)]
Got ✗ [doc-001-vacation(0.61), doc-005(0.55), doc-002(0.44)] — wrong top hit after model swap
Key Assertion
const sources = data.retrievedChunks.map(c => c.source); expect( sources.some(s => s.includes(expectedSource)) ).toBe(true);
✓ Pass
Expected document appears in top-K — retrieval is stable
✗ Fail
Regression after model/chunk change — the right doc is no longer retrieved
🔄
05

Stale Data — Update Propagation

Two-Phase Ingest + Content Assertion
Failure Mode
After a document is updated and re-ingested, the system still returns the old value. The stale embedding was not replaced — both versions now exist in the vector store. The bug is silent: no errors, no crashes, just wrong answers delivered with confidence.

Why This Happens
Vector stores don't automatically replace embeddings on re-ingest. Calling from_documents() with an updated doc ADDS a new vector alongside the old one. Both coexist. Queries may retrieve either version non-deterministically.
How It Works — Two Phases
1
Clear collection → ingest v1 document (e.g. "data retention: 90 days")
2
Query → assert answer contains "90" (baseline confirmed)
3
Clear collection again → ingest v2 document ("data retention: 60 days")
4
Query → assert answer contains "60" AND does not contain "90"

If step 3 skips the clear, both embeddings exist → phase 4 is non-deterministic → you've found the staleness bug.
Concrete Example
Phase 1 — v1 Ingest
Doc v1 "Data must be retained for 90 days"
Query "How long must data be retained?"
Answer ✓ "90 days" → baseline confirmed
Phase 2 — v2 Re-Ingest
Doc v2 "Data must be retained for 60 days" (policy updated)
Query "How long must data be retained?"
Answer ✓ "60 days" + no mention of "90" → propagation works
Answer ✗ "90 days" still returned → stale embedding bug
Key Assertions
// Phase 1 — baseline expect(v1Answer.toLowerCase()).toMatch(/\b90\b/); // Phase 2 — new value present expect(v2Answer.toLowerCase()).toMatch(/\b60\b/); // Phase 2 — old value absent (stale embedding removed) expect(v2Answer).not.toMatch(/\b90\b/);
✓ Pass
After re-ingestion: new value returned, old value gone. Your update pipeline correctly replaces stale embeddings.
✗ Fail
Old value still returned after update. Both embeddings exist in the store — your re-ingest skipped the delete step.
Technique Reference
The 4 evaluation techniques used across all test types
LLM-as-Judge Used in: Faithfulness, Precision
You write a prompt asking Claude to evaluate Claude's output. The judge scores the answer on a specific dimension (grounding, relevance) and returns structured JSON with a score and reasoning.
Why it's used
Full control over what "faithful" or "relevant" means for your domain. Can score nuanced criteria that regex can't. Fast — no extra infra beyond your existing Claude SDK.
Limitation
Scores vary slightly run-to-run (temperature). Use thresholds (≥ 0.85), not exact values. Use a cheaper model (Haiku) as judge — the evaluation task is simpler than generation.
Ragas Used in: Faithfulness
Standardised RAG evaluation library. Uses NLI (Natural Language Inference) internally to check if each answer claim is entailed by the context. Scores are reproducible and comparable across teams.
Why it's used
Widely cited in research. Interviewers and stakeholders recognise it. Good for CI quality gates and benchmarking — gives you a number you can track over time.
Limitation
Runs in Python, requires the Flask microservice. Uses OpenAI internally by default. Slower than LLM-as-Judge. Less customisable for domain-specific criteria.
Phrase Matching Used in: Negative
The simplest possible evaluation: check if the answer contains one of a configured list of refusal phrases. No LLM needed. Fast and deterministic.
Why it's used
For binary yes/no questions ("did the system refuse?") string matching is sufficient and faster. LLM-as-Judge would be overkill and introduce false negatives from judge variability.
Limitation
Depends on the refusal phrase list in rag.config.json. Too strict (exact match) → false failures. Too loose → false passes. Tune the phrase list for your system's actual refusal language.
Deterministic Source Match Used in: Retrieval Unit
No LLM involved. Directly check the source field of retrieved chunks against a pre-defined ground-truth mapping: this query should always return this doc ID.
Why it's used
Retrieval (vector similarity) is deterministic for a given embedding model. The same query should always retrieve the same top chunk. This makes it unit-testable — you know the exact expected output.
When to run
Every time you: swap embedding models, change chunk size/overlap, modify similarity thresholds, or re-index documents. This is your regression guard for the retrieval layer.
